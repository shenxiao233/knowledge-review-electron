# GitHub 与 Gitee 双渠道发布

## Gitee 仓库

在 Gitee 创建仓库：

`shenxiao233/knowledge-review-electron`

如果 Gitee 用户名不同，需要同步修改 `src/main.js` 和 `.github/workflows/release.yml` 中的仓库所有者。

建议创建空仓库，不要自动添加 README、`.gitignore` 或许可证文件，避免首次同步时出现历史不一致。

## GitHub Actions Secret

在 GitHub 仓库的 `Settings > Secrets and variables > Actions` 中新增：

- 名称：`GITEE_TOKEN`
- 值：Gitee 的个人访问令牌

令牌需要具备创建 Release 和上传 Release 附件的权限。未配置令牌时，GitHub 发布仍会正常完成，但不会上传到 Gitee。

## 发布流程

推送版本标签后，工作流会：

1. 构建 Windows 安装包和 `latest.yml`。
2. 创建并发布 GitHub Release。
3. 如果配置了 `GITEE_TOKEN`，通过 Gitee Release API 创建同名 Release，并直接上传安装包、`.blockmap` 和 `latest.yml`。

Gitee 发布不再使用 `git push` 同步源码，只依赖 Release API 和附件上传接口。Gitee 的公开附件接口目前没有稳定可用的分片续传协议，因此工作流使用长超时、重试和已存在附件跳过策略。

## 客户端

设置页的更新渠道支持：

- 自动选择：先检查 GitHub，失败后检查 Gitee。
- GitHub：只使用 GitHub Releases。
- Gitee：先读取 Gitee 最新 Release，再从对应标签下载更新文件。

更新渠道只保存在本机配置中，不会改变卡片、文档和复习记录数据。
